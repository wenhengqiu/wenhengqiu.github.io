## 1. 产品概述

### 1.1 产品定位
一个专注于四大AI领域的行业资讯阅读平台，基于GitHub Pages静态部署，自动展示Info-Getter采集的内容。

| 板块 | 标识 | 内容定位 |
|------|------|----------|
| **人工智能** | ai | 大模型技术前沿 |
| **具身智能** | robotics | 机器人技术突破 |
| **自动驾驶** | autonomous | 智能驾驶进展 |
| **卓驭科技** | zhuoyu | 成行平台追踪 |

### 1.3 页面清单

| 页面 | 文件名 | 功能定位 | 优先级 |
|------|--------|----------|--------|
| **首页** | index.html | 每日资讯、分类筛选、TOP10 | P0 |
| **研究页** | research.html | 四大领域深度内容 | P0 |
| **状态页** | status.html | 系统状态监控 | P1 |
| **时间轴** | timeline.html | 重大事件里程碑 | P3 |

## 2. 文章元信息规范

### 2.1 字段定义与约束

| 字段名 | 类型 | 必填 | 约束规则 | 示例 |
|--------|------|------|----------|------|
| **id** | string | ✅ | 唯一标识，格式：`{日期}-{序号}` 或 `{来源}_{时间戳}_{序号}` | `"2026-03-14-001"` |
| **title** | string | ✅ | 原文标题，最大200字符 | `"OpenAI releases GPT-4 Turbo"` |
| **title_zh** | string | ✅ | 中文标题，最大200字符，Info-Getter翻译 | `"OpenAI发布GPT-4 Turbo"` |
| **summary** | string | ❌ | 原文摘要，最大500字符 | `"OpenAI announced the release of GPT-4 Turbo..."` |
| **summary_zh** | string | ✅ | 中文摘要，最大500字符，Info-Getter翻译 | `"OpenAI宣布发布GPT-4 Turbo..."` |
| **url** | string | ✅ | 原文链接，必须是有效HTTP/HTTPS URL，不允许example.com | `"https://openai.com/blog/gpt-4-turbo"` |
| **source** | object | ✅ | 信息源信息，包含name和type | `{"name": "OpenAI Blog", "type": "official"}` |
| **source.name** | string | ✅ | 信息源名称，最大50字符 | `"OpenAI Blog"` |
| **source.type** | string | ✅ | 信息源类型，枚举值 | `"official"` |
| **category** | string | ✅ | 内容分类，四大领域之一 | `"ai"` |
| **quality_score** | number | ✅ | 质量分数，范围0-1，保留2位小数 | `0.85` |
| **published_at** | string | ✅ | 发布时间，ISO 8601格式 | `"2026-03-14T13:15:50Z"` |
| **translated** | boolean | ✅ | 是否已翻译 | `true` |
| **tags** | array | ❌ | 标签数组，可选 | `["GPT", "大模型", "API"]` |
| **companies** | array | ❌ | 相关公司，可选 | `["OpenAI", "Microsoft"]` |

### 2.2 枚举值定义

**source.type 枚举**:
- `official` - 官方博客/官网
- `tech_media` - 科技媒体
- `research` - 学术论文
- `social` - 社交媒体

**category 枚举**:
- `ai` - 人工智能/大模型
- `robotics` - 具身智能/机器人
- `autonomous` - 自动驾驶
- `zhuoyu` - 卓驭科技

### 2.3 数据格式示例

```json
{
  "id": "2026-03-14-001",
  "title": "OpenAI releases GPT-4 Turbo with 128K context",
  "title_zh": "OpenAI发布GPT-4 Turbo，支持128K上下文",
  "summary": "OpenAI announced the release of GPT-4 Turbo...",
  "summary_zh": "OpenAI宣布发布GPT-4 Turbo，支持128K上下文窗口...",
  "url": "https://openai.com/blog/gpt-4-turbo",
  "source": {
    "name": "OpenAI Blog",
    "type": "official"
  },
  "category": "ai",
  "quality_score": 0.95,
  "published_at": "2026-03-14T13:15:50Z",
  "translated": true,
  "tags": ["GPT-4", "大模型", "API"],
  "companies": ["OpenAI"]
}
```

### 2.4 数据验证规则

| 验证项 | 规则 | 错误处理 |
|--------|------|----------|
| id唯一性 | 同一文件内id不能重复 | 去重，保留第一个 |
| url有效性 | 必须以http://或https://开头 | 丢弃无效URL的文章 |
| url黑名单 | 不允许example.com/localhost | 丢弃黑名单URL |
| quality_score范围 | 0.0 - 1.0 | 超出范围截断到边界 |
| published_at格式 | 必须符合ISO 8601 | 解析失败使用当前时间 |
| category合法性 | 必须是四大领域之一 | 非法分类标记为"other" |
| 必填字段 | id/title/title_zh/url/source/category/quality_score/published_at | 缺少必填字段丢弃文章 |

### 2.5 文件存储规范

**文件路径**: `data/articles/research/{category}.json`

**文件格式**:
```json
[
  {文章对象1},
  {文章对象2},
  ...
]
```

**编码**: UTF-8

**换行**: LF (Unix风格)

**缩进**: 2个空格

**排序**: 按published_at倒序，最新在前

```
└─────────┘    └─────────┘    └──────┘    └─────────────┘    └──┬───┘    └──────────┘    └──┬───┘
                                                                  │                          │
                                                                  └──────────────────────────┘
                                                                                             ↓
┌──────────┐    ┌──────────┐    ┌─────────┐                                              ┌──────────┐
│ 正式发布 │ ← │ 试用通过 │ ← │ 试用验证 │                                              │ 试用验证 │
└──────────┘    └──────────┘    └────┬────┘                                              └────┬────┘
                                     │                                                        │
                                     └────────────────────────────────────────────────────────┘
                                     试用不通过 → 反馈收集 → 返回需求评审/开发实现/测试验证
```

### 5.2 各阶段规范（全Agent自动化）

#### 阶段1: 需求评审
- **执行Agent**: Agent-Product-Manager
- **输入**: PRD文档
- **处理**: 解析功能点、优先级、验收标准
- **输出**: 确认后的开发任务清单
- **通过标准**: 任务清单明确，无歧义

#### 阶段2: 开发实现
- **执行Agent**: Agent-Developer
- **输入**: 任务清单 + PRD规范
- **开发规范**:
  - 按功能模块开发，每个功能独立提交
  - 提交信息格式: `[功能ID] 功能描述`
  - 示例: `[HP-001] 实现今日TOP10展示`
- **代码规范**:
  - HTML语义化标签
  - CSS使用变量，便于主题统一
  - JavaScript模块化，避免全局污染
- **输出**: 功能代码 + 自测报告

#### 阶段3: 自测
- **执行Agent**: Agent-Developer（自检）
- **自测清单**:
  - [ ] 功能是否正常实现
  - [ ] PRD中的测试用例是否通过
  - [ ] 控制台无报错
  - [ ] 移动端是否正常
  - [ ] 性能是否达标
- **输出**: 自测通过/不通过 + 问题列表
- **不通过**: 返回阶段2修复

#### 阶段4: 测试Agent验证 【关键】
- **执行Agent**: Agent-Tester
- **触发条件**: 自测通过后自动触发
- **测试内容**:
  1. 按PRD测试用例逐项验证
  2. 用户场景模拟测试
  3. 兼容性测试
  4. 性能测试
- **输出**: 测试报告（通过/不通过 + 问题列表 + 严重程度）
- **通过标准**: 所有P0测试用例通过，无严重bug
- **不通过**: 自动返回阶段5

#### 阶段5: 修复
- **执行Agent**: Agent-Developer
- **输入**: 测试报告
- **处理**: 针对问题逐项修复
- **修复后**: 自动重新自测（阶段3）→ 测试验证（阶段4）
- **循环**: 直至测试通过

#### 阶段6: 验收通过
- **执行Agent**: Agent-Product-Manager
- **输入**: 测试报告 + 功能演示
- **验收内容**:
  - 功能符合PRD描述
  - 测试Agent报告通过
  - 代码质量达标
- **输出**: 验收通过/不通过
- **不通过**: 返回阶段2重新开发

#### 阶段7: 上线
- **执行Agent**: Agent-DevOps
- **自动执行**:
  1. 合并到master分支
  2. 推送到GitHub
  3. 等待GitHub Pages部署（2-3分钟）
  4. 线上验证（访问生产环境，验证核心功能）
- **输出**: 上线成功/失败
- **成功后**: 自动触发阶段8

#### 阶段8: 试用验证
- **执行Agent**: Agent-User-Tester（模拟真实用户）
- **试用周期**: 1-3天
- **试用内容**:
  - 按真实场景使用功能
  - 记录问题反馈
  - 评估体验满意度
- **输出**: 试用报告（通过/不通过 + 问题列表 + 体验评分）
- **通过**: 自动进入阶段10
- **不通过**: 自动进入阶段9

#### 阶段9: 问题反馈与迭代
- **执行Agent**: Agent-Product-Manager
- **输入**: 试用报告
- **自动处理**:
  - 分析问题根因
  - 确定回退阶段（需求/开发/测试）
  - 生成修复任务
- **自动回退**:
  - 需求问题 → 返回阶段1
  - 技术问题 → 返回阶段2
  - 测试覆盖问题 → 返回阶段4
- **循环**: 重新走完整流程，直至试用通过

#### 阶段10: 正式发布
- **执行Agent**: Agent-DevOps
- **自动执行**:
  - 标记版本号（git tag）
  - 更新CHANGELOG.md
  - 生成发布说明
- **发布后监控（3天）**:
  - Agent-Monitor持续监控
  - 收集用户反馈
  - 发现问题自动进入阶段9

```
│ -Manager        │←────│                 │←────│                 │
│ (需求/验收)     │     │ (开发/修复)     │     │ (测试/验证)     │
└─────────────────┘     └─────────────────┘     └─────────────────┘
         ↑                                            │
         └────────────────────────────────────────────┘
                           
┌─────────────────┐     ┌─────────────────┐
│ Agent-User      │────→│ Agent-DevOps    │
```

### 5.4 自动化触发机制

| 阶段 | 触发条件 | 执行Agent | 输出物 |
|------|----------|-----------|--------|
| 需求评审 | PRD更新 | Agent-Product-Manager | 任务清单 |
| 开发实现 | 任务分配 | Agent-Developer | 代码+自测报告 |
| 自测 | 开发完成 | Agent-Developer | 自测结果 |
| 测试验证 | 自测通过 | Agent-Tester | 测试报告 |
| 修复 | 测试不通过 | Agent-Developer | 修复代码 |
| 验收 | 测试通过 | Agent-Product-Manager | 验收结果 |

### 5.5 人工介入点

虽然全流程自动化，但以下情况需要人工决策：

1. **PRD重大变更**: 需人工确认后更新
2. **架构调整**: 技术架构变更需人工评审
3. **发布审批**: 大版本发布前人工最终确认
4. **异常处理**: Agent无法处理的异常情况

### 5.3 测试Agent规范

**测试Agent触发时机**:
- 每个功能开发完成后
- 修复bug后
- 上线前最终验证

**测试Agent任务模板**:
```
你是一名严格的质量测试工程师，正在测试功能 [功能ID]。

测试依据: PRD v6.1 中的测试用例
测试环境: https://wenhengqiu.github.io/

请执行以下测试：
1. 基础功能测试（按PRD测试用例）
2. 用户场景测试（模拟真实用户操作）
3. 兼容性测试（Chrome/Safari/iPhone）
4. 性能测试（加载时间、响应速度）

输出格式：
## 测试结果
- 通过/不通过
- 问题列表（如有）
- 严重程度（致命/严重/一般/建议）
```

### 5.4 问题分级

| 级别 | 定义 | 处理方式 |
|------|------|----------|
| **致命** | 功能完全不可用，或导致数据丢失 | 必须修复，否则不能上线 |
| **严重** | 核心功能异常，或严重影响体验 | 必须修复，否则不能上线 |
| **一般** | 非核心功能异常，或有替代方案 | 建议修复，可延期 |
| **建议** | 体验优化建议，非bug | 可记录，后续版本考虑 |
